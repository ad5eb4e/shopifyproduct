# Shopify 代理中台 — 需求文档

## 1. 系统定位

一个运行在本地 Mac mini 上的 **Shopify 代理中台**，统一管理多个 Shopify 店铺的凭证、代理和 API 访问，为各类业务应用提供标准化的 Shopify 数据通道。

**核心价值：** 业务应用不需要关心"怎么连 Shopify"，只需要告诉中台"要对哪个店铺做什么操作"。

```
┌──────────────────────────────────────────────────────────────────┐
│                         业务应用层                                 │
│                                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │
│  │ 产品同步  │ │ 合规检查  │ │ 合规回写  │ │ 产品复制  │ │ 报表…  │ │
│  │ Shopify→  │ │ 飞书→    │ │ 飞书→    │ │ A站→B站  │ │        │ │
│  │ 飞书     │ │ 检查→飞书 │ │ Shopify  │ │          │ │        │ │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ └───┬────┘ │
│       │            │            │            │            │      │
│  ─ ─ ─│─ ─ ─ ─ ─ ─│─ ─ 飞书多维表（数据总线）─│─ ─ ─ ─ ─ ─│─ ─  │
│       │            │            │            │            │      │
├───────┼────────────┼────────────┼────────────┼────────────┼──────┤
│       ▼            ▼            ▼            ▼            ▼      │
│  ┌──────────────────────────────────────────────────────────────┐│
│  │                     Shopify 代理中台                          ││
│  │                                                              ││
│  │  ● 店铺管理（凭证、代理、认证方式）                             ││
│  │  ● Shopify API 统一客户端（读产品、写产品、读集合…）            ││
│  │  ● 每店独立代理、限速、OAuth 自动续期                           ││
│  │  ● 飞书读写能力（lark-cli 封装）                               ││
│  │  ● Web 管理后台（配置店铺、注册应用、查看状态）                  ││
│  │  ● 定时调度（按应用注册的任务触发）                             ││
│  └──────────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────────┘
```

**应用间解耦原则：**
- 每个应用是独立目录，有自己的代码和配置
- 应用之间**不互相调用**，只通过飞书多维表交换数据
- 应用只依赖中台提供的能力（ShopifyClient、LarkClient、Scheduler）
- 新增应用不修改任何已有应用的代码

## 2. 中台功能

### 2.1 店铺管理

通过 Web 管理后台统一管理所有 Shopify 店铺连接：

**管理界面：内联表格，直接编辑保存**

| 店铺名称 | 店铺标识 | 前端域名 | 认证方式 | 凭证 | 代理IP | 已启用应用 | 操作 |
|---------|---------|---------|---------|------|-------|-----------|------|
| US-01 | us-store-01 | www.us-store.com | Token | shpat_**** | socks5://1.2.3.4:1080 | 同步 / 合规 / 回写 | 保存 / 删除 |
| AU-02 | au-store-02 | www.au-store.com | OAuth | ID:**** | socks5://5.6.7.8:1080 | 同步 / 回写 | 保存 / 删除 |
| | | | | | | | +新增 |

**店铺配置字段（中台级别）：**

| 字段 | 说明 |
|------|------|
| 店铺名称 | 自定义名称，如 US-01、AU-02 |
| 店铺标识 | Shopify 后台域名前缀，如 `us-store-01` |
| 前端域名 | 客户访问的网站地址 |
| 认证方式 | Token 或 OAuth |
| Access Token / Client ID / Client Secret | 根据认证方式填写 |
| 代理IP | 该店铺专用的代理地址 |
| 飞书 Table ID | 该店铺在多维表中对应的表格 ID |

**通用操作：**
- 「测试连接」— 验证凭证 + 代理可达
- 「查看日志」— 查看该店铺所有应用的操作记录（可按应用筛选）

### 2.2 中台提供的能力

中台对上层应用暴露以下标准化能力（内部 Python API，非 HTTP）：

**Shopify 操作：**

| 操作 | 说明 | 需要 Token |
|------|------|-----------|
| `list_products(store)` | 获取店铺所有产品列表 | 否 |
| `get_product(store, handle)` | 获取单个产品详情 | 否 |
| `get_product_seo(store, handle)` | 获取产品 SEO 信息 | 否 |
| `get_product_admin(store, id)` | 获取产品完整数据 | 是 |
| `get_product_collections(store, id)` | 获取产品所属集合 | 是 |
| `update_product(store, id, data)` | 更新产品 | 是 |
| `update_product_seo(store, id, title, desc)` | 更新 SEO | 是 |

**飞书操作：**

| 操作 | 说明 |
|------|------|
| `lark.list_records(table_id, filter)` | 读取多维表记录 |
| `lark.batch_create(table_id, records)` | 批量创建记录 |
| `lark.batch_update(table_id, records)` | 批量更新记录 |
| `lark.query(table_id, dsl)` | DSL 条件查询 |

**调度：**

| 操作 | 说明 |
|------|------|
| `scheduler.register(store, app, tasks)` | 为店铺注册应用定时任务 |
| `scheduler.unregister(store, app)` | 注销任务 |

所有 Shopify 操作自动走该店铺的代理、自动处理认证、自动限速。

### 2.3 应用注册机制

每个应用需要声明：

| 声明项 | 说明 | 示例 |
|--------|------|------|
| `app_name` | 唯一标识 | `"sync"` / `"compliance"` / `"writeback"` |
| `config_schema` | 该应用需要的配置字段 | `{"light_cron": "0 3 * * *", "deep_cycle": 3}` |
| `scheduled_tasks` | 需要注册的定时任务 | `[{"func": light_sync, "cron": "0 3 * * *"}]` |
| `routes` | 应用专属的 Web 路由（可选） | `POST /api/stores/<id>/sync` |

中台在启动时自动发现并加载 `apps/` 目录下所有已注册的应用。

### 2.4 Shopify API 认证

#### 方式一：Access Token（2026.1.1 前创建的 Custom App）

1. 登录 Shopify 后台
2. Settings > Apps > Develop apps > Create an app
3. Configure Admin API scopes：`read_products`, `write_products`, `read_product_listings`
4. Install app → 复制 Admin API access token（只显示一次）

#### 方式二：OAuth Client Credentials（2026.1.1 后新店铺）

1. [Shopify Partners Dashboard](https://partners.shopify.com) > Create app
2. 配置 scopes，获取 Client ID + Client Secret
3. 安装到店铺，系统自动获取 token（24小时过期自动续期）

## 3. 已实现应用

### 3.1 产品同步（app: sync）

**职责：** 从 Shopify 拉取产品数据 → 写入飞书多维表。只负责"拉"，不负责"检查"和"回写"。

```
系统按设定频率拉取产品数据 → 写入飞书多维表 → 结束
```

**应用配置：**

| 字段 | 说明 |
|------|------|
| 轻量扫描频率 | 检测新产品和基础字段变化，可选：每小时 / 每6小时 / 每天 + 时间 |
| 深度扫描周期 | 几天循环扫完所有产品的SEO/图片Alt等，可选：1-7天 |
| 深度扫描时间 | 每天执行深度扫描批次的时间 |

**功能：**
1. 轻量扫描：通过前端 `/products.json` 检测新产品、基础字段变化
2. 深度扫描：逐个产品检测 SEO、图片 Alt（分批循环）
3. 新产品入库时标记 `合规状态 = "待检查"`
4. 已有产品字段变化时重置 `合规状态 = "待检查"`

**不做的事：** 不检查合规，不回写 Shopify。

### 3.2 合规检查（app: compliance）

**职责：** 从飞书多维表读取待检查产品 → 检测风控风险 → 自动改写 → 将结果写回飞书多维表。只负责"检查和改写"，不直接操作 Shopify。

```
定时扫描飞书中"合规状态=待检查"的记录
        ↓
读取产品标题、描述、标签等 10 个可扫描字段
        ↓
按照合规规则检测是否命中风控关键词
        ↓
├── 全部合规 → 合规状态 = "合规"
└── 需要改写 → 自动改写 → 写入改写字段 → 合规状态 = "已改写"
```

**应用配置：**

| 字段 | 说明 |
|------|------|
| 检查频率 | 多久检查一次飞书中的待检查记录，如每小时 |
| 检查规则 | 使用 COMPLIANCE_PROMPT.md 中的风控规则 |

**检查范围：** 产品标题、描述、产品类型、标签、SEO标题、SEO描述、Handle、图片Alt、规格名称、所属集合（共 10 个字段）。

**不做的事：** 不拉 Shopify 数据，不回写 Shopify。只和飞书多维表交互。

### 3.3 合规回写（app: writeback）

**职责：** 从飞书多维表读取已改写的记录 → 通过中台写回 Shopify。只负责"回写"，不负责检查。

```
定时扫描飞书中"合规状态=已改写 且 回写状态≠已回写"的记录
        ↓
通过中台 ShopifyClient 将改写内容更新到对应店铺
        ↓
├── 成功 → 回写状态 = "已回写"，原始字段同步更新
└── 失败 → 回写状态 = "回写失败"
```

**应用配置：**

| 字段 | 说明 |
|------|------|
| 回写频率 | 多久检查一次，默认每小时 |

**不做的事：** 不拉数据，不检查合规。只读飞书 → 写 Shopify。

### 3.4 三个应用的协作关系

```
Shopify ──→ [产品同步] ──→ 飞书多维表 ──→ [合规检查] ──→ 飞书多维表 ──→ [合规回写] ──→ Shopify
                              ↑                                                │
                              └────────────── 原始字段同步更新 ─────────────────┘
```

**关键解耦点：**
- 三个应用只通过飞书多维表的**字段状态**协作，代码完全独立
- 停掉合规检查 → 同步照常工作，只是没人检查
- 停掉回写 → 检查照常，只是改写结果不回 Shopify
- 任何一个应用可以独立升级、重启、配置，不影响其他

### 3.5 飞书多维表

#### 表格结构

默认所有店铺共用一张多维表，通过"店铺名称"区分。

#### 字段说明

**基础信息（sync 应用写入）：**

| 字段 | 说明 |
|------|------|
| 店铺名称 | 来自哪个店铺 |
| Shopify产品ID | 产品唯一编号 |
| 产品状态 | active / draft / archived |
| 价格 | 产品价格 |
| SKU | 产品SKU编码 |
| 前端链接 | 产品在网站上的访问地址 |
| 产品图片 | 所有产品主图URL，换行分隔 |
| Variant图片 | 每行格式：规格名 竖线 图片URL |

**风控可扫描字段（sync 写入原始值，compliance 写入改写值）：**

| 原始字段 | 改写字段 | 写入方 |
|---------|---------|-------|
| 产品标题 | 改写标题 | sync / compliance |
| 原始描述HTML | 改写描述 | sync / compliance |
| 原始描述纯文本 | — | sync |
| 产品类型 | 改写产品类型 | sync / compliance |
| 产品标签 | 改写标签 | sync / compliance |
| SEO标题 | 改写SEO标题 | sync / compliance |
| SEO描述 | 改写SEO描述 | sync / compliance |
| Handle | 改写Handle | sync / compliance |
| 图片Alt文本 | 改写图片Alt | sync / compliance |
| 规格名称 | 改写规格名称 | sync / compliance |
| 所属集合 | — | sync |

**状态字段（应用间协调的枢纽）：**

| 字段 | 写入方 | 说明 |
|------|-------|------|
| 合规状态 | sync 设"待检查" / compliance 设"合规"或"已改写" | 三态流转 |
| 回写状态 | writeback 设"已回写"或"回写失败" | 三态流转 |
| 最后更新时间 | 各应用更新 | |

#### 状态流转

```
[sync] 新产品入库 / 字段变化 → 合规状态: 待检查
                                    ↓
[compliance] 检查全部合规          → 合规状态: 合规（流程结束）
[compliance] 需要改写 → 写入改写值 → 合规状态: 已改写
                                    ↓
[writeback] 读取改写值 → 回写 Shopify → 回写状态: 已回写
                                      → 原始字段同步更新（防下次误判）
[writeback] 回写失败               → 回写状态: 回写失败
```

## 4. 未来应用规划（待定）

以下应用均基于中台能力构建，与现有应用完全独立：

### 4.1 产品复制（app: product_copy）

从 A 店铺读取产品 → 写入 B 店铺。通过中台的 ShopifyClient 分别操作两个店铺。

### 4.3 新增应用的步骤

1. 在 `apps/` 下创建新目录
2. 实现应用逻辑，只调用中台提供的能力
3. 声明 `app_name`、`config_schema`、`scheduled_tasks`
4. 中台自动发现并加载

无需修改中台代码或其他应用代码。

## 5. 注意事项

1. **后台访问安全**：管理后台需要登录，首次部署时设置密码。
2. **API 凭证安全**：Token / Secret 加密存储，页面星号显示。
3. **OAuth 自动续期**：Token 24 小时过期，系统自动重新获取。
4. **代理IP**：每个店铺独立代理，互不关联。
5. **应用独立性**：每个应用可以独立启停、配置，不影响其他应用。
6. **扫描频率**：轻量扫描每天1-3次；深度扫描几天循环扫完。
7. **改写格式**：改写描述用 HTML；标签用逗号分隔；图片Alt和规格用换行分隔。
8. **只改需要改的**：不需要改写的字段留空。
